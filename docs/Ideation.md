# Poker-Mate 아이디에이션 (Ideation)

이 문서는 Poker-Mate 프로젝트의 기획 배경, 핵심 문제 정의, 그리고 해결 방안을 도출하는 과정을 담고 있습니다.

## 1. 기획 동기 (Motivation)
**"왜 포커 AI인가?"**
*   **기존 툴의 한계:** 시중의 '포커 계산기(Solver)'들은 정확하지만 너무 딱딱하고 어렵습니다. 초보자가 사용하기에는 진입 장벽이 높습니다.
*   **AI의 가능성:** LLM은 대화가 가능하지만, 게임 룰을 완벽히 이해하지 못하거나 계산이 느립니다.
*   **결론:** "계산기의 정확성 + 챗봇의 친근함"을 결합하면 어떨까? 게임 도중 실시간으로 조언해주고, 멘탈까지 케어해주는 '파트너'를 만들어보자.

## 2. 핵심 도전 과제와 해결책 (Core Challenges & Solutions)

### 도전 1: 속도 (Speed)
*   **문제:** 홀덤은 턴 제한 시간이 짧습니다(보통 15~30초). 일반적인 LLM 응답(3~5초 소요)을 기다리다가는 턴이 넘어갑니다.
*   **해결책:**
    *   **SSE (Server-Sent Events):** 사용자가 기다리지 않게, AI가 생각하는 대로 토큰을 실시간으로 쏘아줍니다. Polling보다 효율적입니다.
    *   **Redis Caching (단기 기억):** 이전 대화 맥락(Context)을 매번 DB에서 긁어오지 않고, 인메모리(Redis)에서 1ms 안에 조회합니다.

### 도전 2: 정확성 (Accuracy)
*   **문제:** LLM은 숫자 계산에 약하고, 없는 족보를 만들어내는 '할루시네이션'이 발생할 수 있습니다. (예: "스트레이트 플러시가 원 페어한테 졌어요" 같은 헛소리 방지)
*   **해결책:**
    *   **Context Injection:** AI가 답변을 생성하기 전에, 미리 정의된 정확한 **'포커 확률표(JSON)'** 데이터를 프롬프트에 강제로 주입합니다. AI는 이 데이터를 '참고서'처럼 보고 정확한 답변을 내놓을 수 있습니다.

## 3. 페르소나 설계 (Persona Design)
**"상황에 맞는 다른 목소리"**
사용자는 때로는 냉철한 분석이 필요하고, 때로는 따뜻한 위로가 필요합니다. 하나의 성격으로는 부족합니다.
*   **전략가 (Strategist):** "현재 팟 오즈는 25%입니다. 콜 하는 것이 수학적으로 이득입니다." (이성적, 건조함)
*   **동반자 (Companion):** "아이고, 이번엔 운이 진짜 없었네요. 그래도 플레이 과정은 좋았어요! 툭 털고 다음 판 집중해봐요." (감성적, 친근함)
*   **구현:** 사용자의 선택이나 상황 변수에 따라 Spring Boot 서버가 동적으로 다른 `System Prompt`를 갈아끼웁니다.

## 4. 기술 스택 선정 배경 (Tech Stack)
*   **Java 17 & Spring Boot:** 엔터프라이즈급 안정성이 필요했습니다. 비동기 처리(`Flux`)와 데이터 처리에 강력합니다.
*   **Redis:** 실시간성을 위해 선택했습니다. 단순 캐시를 넘어 대화 세션의 '단기 기억 저장소'로 활용합니다.
*   **MySQL:** 사용자의 플레이 스타일 데이터는 영구적으로 보존되어야 하므로, 신뢰성 있는 RDBMS를 선택했습니다.
